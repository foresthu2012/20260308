# 【永久强制规范】Niushop V6 插件开发专用指令
从现在开始，你必须**完整记住、严格执行**以下所有规范，**任何一条都不允许违反**，违反则立即停止生成代码。

---

## 一、基础开发环境（固定不变）
- 框架：Niushop V6 / Niucloud-admin
- 后端：ThinkPHP 8.x
- 前端：Vue3 + Element Plus
- 移动端：uni-app
- 运行模式：**仅单店铺，不处理任何多店铺、SaaS、多租户逻辑**
- 开发方式：**纯插件开发，绝对禁止修改框架核心文件**

---

## 二、数据库表前缀（绝对强制约束）
- 系统默认表前缀：**ns_**
- **严禁以任何形式修改、拼接、自定义表前缀**
- **严禁在代码中出现除 ns_ 以外的任何表前缀**
- **严禁手写带前缀的完整表名**
- 模型中只允许写表名主体，框架会自动拼接前缀
  ✅ 正确写法：`protected $name = 'order';`
  ❌ 错误写法：`protected $table = 'yx_order';` / `protected $table = 'ns_order';`

---

## 三、插件标准目录结构（必须严格遵守）
```
插件标识/
├── admin/                  # 后台前端（Vue3 + Element Plus）
│   ├── api/                # 前端接口请求定义
│   └── views/              # 页面：list、add、detail
├── app/                    # 后端业务（ThinkPHP 8）
│   ├── adminapi/           # 后台接口控制器
│   ├── api/                # 前台/会员端接口
│   ├── model/              # 数据模型
│   └── service/            # 业务逻辑层（所有业务必须写在此层）
├── uni-app/                # 小程序/H5 端
│   ├── api/                # 小程序接口
│   └── pages/              # 小程序页面
├── sql/                    # 数据库安装、升级脚本
└── info.json               # 插件元数据配置文件
```

## 七、路由规范
- 菜单路由必须以 `addon/插件标识/` 开头
- 页面传参使用 query 传递 member_id
- 后台接口路由统一放在 `app/adminapi/route/route.php`

---

## 八、接口响应规范（统一格式）
- 成功：`return success($data, '操作成功');`
- 失败：`return error('错误提示信息');`
- 获取参数：`$this->request->param('字段名');`

---

## 九、业务强制规则
1. 仅支持单店铺，禁止任何多店铺逻辑
2. 选择会员后必须传递 member_id
3. 订单列表/选择器必须按 member_id 筛选
4. 上传必须使用官方上传组件
5. 业务逻辑必须写在 service 层
6. 插件仅允许操作自身表，禁止操作系统表

## 额外编码规范（必须遵守）
• 代码保持简洁、精简，禁止冗余、重复定义
• 严格遵循 ThinkPHP / Niushop 官方原生开发规范
• 模型文件只保留必要属性，不新增多余字段、多余配置
• 所有数据库表名均通过“模型$name属性 + 系统默认前缀ns_”自动生成，禁止任何形式硬编码完整表名（包括旧前缀、ns_前缀）
“开发前务必核对 Niushop 核心表的字段名，比如订单状态字段为status而非order_status，所有涉及订单状态的查询、赋值都必须使用正确字段名”
“禁止在 SQL 查询中使用数据库不存在的字段名，所有前端展示需要的衍生字段（如状态文本）必须通过代码映射生成，禁止直接写入 SQL 的 field 列表。” 这样能从根源上避免反复出现 1054 错误。
“每次生成数据库操作代码后，必须和 shop_order 表的实际字段做比对”

## 具体执行步骤
1. 全局搜索插件目录下所有 PHP、SQL、JS、XML 文件，定位所有硬编码数据表前缀的代码，生成变更清单。
2. 重点检查订单、会员、关联查询等模块，确保 SQL 语句、模型定义、JOIN、子查询全部统一为官方规范表前缀。
3. 修正模型文件：仅保留 `protected $name = '表名主体';`（唯一标准），删除所有 `protected $table` 硬编码表名的写法，确保表名规范、无冲突、无冗余。
4. 检查配置文件，确保表前缀统一，无二次硬编码。
5. 替换完成后，验证数据表结构一致性，确保无字段、索引、外键异常。
6. 对核心功能（列表查询、详情、关联数据、会员相关）做功能测试，确保数据正常读取。
7. 清除缓存，测试页面加载、数据获取、操作提交是否正常，无 SQL 错误、无表不存在报错。
8. 输出：变更清单、修改说明、测试结果。
